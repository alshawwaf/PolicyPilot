1 Describe the access

Pick a layer, then state the request as source → destination : service. Optionally attach a ticket id for the audit trail.

Check Point isn’t only TCP/UDP. Port-based (TCP/UDP/SCTP) takes a port; ICMP, RPC, DCE-RPC, GTP, Compound/Citrix TCP, “Other” (an IP protocol) and a services group are matched by name against the real predefined object; Application/Site matches an app. The engine reasons about each kind on its own, so distinct protocols never falsely overlap a port.

Source and destination aren’t only IPs either. Switch either type to a Domain (alshawwaf.ca, or .alshawwaf.ca for sub-domains), Access role, Dynamic object, Updatable object or Security zone — so you can ask, say, “does host 10.1.1.222 have access to the domain alshawwaf.ca?”. Each object kind is its own match space, so the engine reasons in identity terms — a domain matches a rule’s domain object (exact or a parent), never an IP cell. Domains and dynamic objects are created if missing; access-roles, security-zones and updatable-objects are reused (defined in Identity Awareness, topology, or Check Point’s repository).

2 Recent changes & rollback

Every change published to this server — from this form, the agent, or the webhook — is recorded with its exact inverse. Roll one back to undo just that change by first-match: PolicyPilot deletes the rule it added, re-enables the rule it disabled, or removes the object it widened in — never a full-database revision rollback. A revoke that disabled a rule can also be finalized by deleting that rule. If a rule was removed out-of-band, the rollback completes anyway. Dry-runs aren’t listed; objects a change created are left in place. Use ✕ to drop an entry from this list, or Clear resolved to tidy the done ones — neither touches live policy.

How it decides

Every request runs the same first-match, top-down walk of the rulebase — always aiming at an actionable result, never a hand-off to a human. The rules it follows, top to bottom:

  • Reuse first. If a rule already permits the traffic, it’s a no-op — just attach it to the ticket.
  • Widen only on an exact match. Extend an existing rule only when it already matches the request in the other two columns — so it never grants access nobody asked for.
  • Otherwise create least-privilege. A fresh rule, placed above a blocking deny in the path so the requested access actually takes effect.
  • Match by kind. Each source / destination is compared in its own space — an IP / CIDR by address, or a typed object (domain, access-role, dynamic, updatable, security-zone) by identity — so a domain is matched against a domain object, never confused with an IP.
  • Note, don’t stop. A rule it can’t fully resolve (an updatable feed, an unresolvable cell, a conditional or inline-layer rule) is flagged “possible match — review later” and the walk continues; the new rule goes below it, so it can’t leap over a possible block.
  • Dynamic layers are out of scope. A rule applying a dynamic layer (sk182252) is skipped entirely — its contents are managed out-of-band (Gaia pushes them straight to the gateway), so they’re not handled on the management side.
  • Only a broken request stops it. The rare exception — if the request itself can’t be resolved (an empty service, or a typed object that doesn’t exist), it’s flagged for you to fix and retry.
