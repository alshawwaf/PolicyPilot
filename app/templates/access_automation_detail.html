Describe the access

Pick a layer, then state the request as source → destination : service. Optionally attach a ticket id for the audit trail.

Check Point isn’t only TCP/UDP. Port-based (TCP/UDP/SCTP) takes a port; ICMP, RPC, DCE-RPC, GTP, Compound/Citrix TCP, “Other” (an IP protocol) and a services group are matched by name against the real predefined object; Application/Site matches an app. The engine reasons about each kind on its own, so distinct protocols never falsely overlap a port.

Source and destination aren’t only IPs either. Switch either type to a Domain (alshawwaf.ca, or .alshawwaf.ca for sub-domains), Access role, Dynamic object, Updatable object or Security zone — so you can ask, say, “does host 10.1.1.222 have access to the domain alshawwaf.ca?”. Each object kind is its own match space, so the engine reasons in identity terms — a domain matches a rule’s domain object (exact or a parent), never an IP cell. Domains and dynamic objects are created if missing; access-roles, security-zones and updatable-objects are reused (defined in Identity Awareness, topology, or Check Point’s repository).

Advanced options — action, time, content, limit, targets (all optional)
Action settings — Accept / Ask / Inform
UserCheck — Ask / Inform prompt · Drop / Reject block message

Each name resolves to a REAL Check Point object on Preview — a typo fails loud, nothing is written. Setting any advanced field makes the request restricted: the engine CREATEs a precise rule above a broad Accept so the new condition takes effect (never a false reuse). Time / content / limit / VPN / install-on objects are reused (defined in the SMS), never created.
