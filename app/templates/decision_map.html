Decision map

The principles PolicyPilot follows to turn an access request into the minimal correct change — always aiming at an actionable result, never a hand-off to a human. It runs the same first-match, top-down walk of the rulebase every time. To watch the flow node-by-node, open the Decision tree.

Decision map: PolicyPilot resolves an access request with a strict priority order. First it tries to reuse an existing permit (a no-op). Otherwise it widens an existing rule, but only on an exact match in the other two columns. Otherwise it creates a least-privilege rule placed above any blocking deny. Sources and destinations are matched by kind. Rules it cannot fully resolve are noted, not stopped, and the new rule is placed below them. Rules applying a dynamic layer are out of scope. The only thing that stops the engine is a broken request. See the Decision tree to step through the exact path for any outcome.

Priority order

What it tries, in order

A request flows left to right through the same three stages every time. The engine stops at the first one that fits — the smallest change that satisfies the request always wins.

Match by kind

Resolve

Each source / destination is compared in its own space — an IP / CIDR by address, or a typed object (domain, access-role, dynamic, updatable, security-zone) by identity — so a domain is matched against a domain object, never confused with an IP.

Note, don't stop

Note

A rule it can't fully resolve (an updatable feed, an unresolvable cell, a conditional or inline-layer rule) is flagged “possible match — review later” and the walk continues. Because it might still be a deny, the new rule is placed below it — so first-match keeps that unresolved rule in effect and the new allow can never leap over a possible block.

Dynamic layers are out of scope

Skip

A rule applying a dynamic layer (sk182252) is skipped entirely — its contents are managed out-of-band (Gaia pushes them straight to the gateway), so they're not handled on the management side.

Only a broken request stops it

Fix

The rare exception — if the request itself can't be resolved (an empty service, or a typed object that doesn't exist), it's flagged for you to fix and retry.
