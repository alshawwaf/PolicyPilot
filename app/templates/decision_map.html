Decision map

The principles PolicyPilot follows to turn an access request into the minimal correct change — always aiming at an actionable result, never a hand-off to a human. It runs the same first-match, top-down walk of the rulebase every time. To watch the flow node-by-node, open the Decision tree.

Match by kind

Each source / destination is compared in its own space — an IP / CIDR by address, or a typed object (domain, access-role, dynamic, updatable, security-zone) by identity — so a domain is matched against a domain object, never confused with an IP.

Note, don’t stop

A rule it can’t fully resolve (an updatable feed, an unresolvable cell, a conditional or inline-layer rule) is flagged “possible match — review later” and the walk continues; the new rule goes below it, so it can’t leap over a possible block.

Dynamic layers are out of scope

A rule applying a dynamic layer (sk182252) is skipped entirely — its contents are managed out-of-band (Gaia pushes them straight to the gateway), so they’re not handled on the management side.

Only a broken request stops it

The rare exception — if the request itself can’t be resolved (an empty service, or a typed object that doesn’t exist), it’s flagged for you to fix and retry.
