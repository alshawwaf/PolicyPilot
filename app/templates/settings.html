How an agent is authorized — two independent gates

Publish decides whether an agent may commit to live policy at all; Autopilot decides whether it may do so in one turn without you saying “publish”. Autopilot rides on Publish (with Publish off, nothing commits). Both are separate from the engine’s Behavior profile on Access automation logic, which only shapes the change.

If you want…PublishAutopilotWhat the agent does
Look, don’t touch off— Always previews / dry-runs; a publish is refused.
Preview, then you approve onoff Dry-runs first; commits only when your message says “publish”.
One sentence → done (demo) onon Resolves, applies and publishes in one turn — no confirmation step.

Shortcut: the ⚡ demo preset on Access automation logic flips both of these on (plus the Aggressive profile) in one click. A read-only API key can never publish regardless of these.

API keys

Keys are stored hashed and shown once at creation — copy it then. Revoking is immediate. (The single shared token in MCP / agent and Ticketing webhook above still works too, as a fallback.)

Best practice: give each client its own named key, set an expiry (default 90 days), and revoke keys that show unused or that you no longer recognise. Keys are least-privilege by scope — an mcp key can't call the REST API and vice-versa — and by access: a read-only key lets an agent preview and read (decide_access, fetch_dynamic_layer, list_*, …) but every write tool refuses, so you can give an LLM look-but-don't-touch access. Writes still need the publish/push gate on top.
